# M8 — Mobile + polish

Executable plan for milestone M8 of `RAPIDLY_ENGINEERING_SUITE_PLAN.md`.
M8 makes the suite usable on tablet for field engineers, adds Web
Push for workflow alerts, and polishes the rough edges accumulated
through M1–M7.

**Read M0–M7 first.** M8 binds against:
- The renamed Markup chamber (M1.4) — most of the tablet UX work lives here.
- M3's viewer (also needs touch nav).
- M5's agent UI (read-only on mobile).
- M6's site-walk node (M8 wires the upload path).
- M7's integrations (mobile UI lists notifications from each).

## Scope decisions (locked)

- **Tablet PWA, phone read-only.** Strategic plan §11/6 resolved: iPad Safari + Android Chrome are primary; phones get a read-only drawer-style UI (view boards, RFIs, notifications; no editing). Justification: real markup needs ≥9" screen; a phone is a notification + camera surface, not an editing surface.
- **Service worker via `@serwist/next`** (formerly next-pwa, modern actively-maintained successor). Already-known-good in this Next.js + App Router setup.
- **Web Push via VAPID.** Standard, no vendor SDK. iOS Safari supports it from 16.4+; Android Chrome from ~50+. No native app.
- **No native app this milestone.** Capacitor / React Native deferred to M9+ if the pilot wants it.

## Scope (5 PRs, ~2 weeks)

## Conventions

- All M8 PRs run on tablet hardware before merge, not just Chrome devtools' device emulator. Note in DoD: which physical device tested.
- The PWA install must work on iOS Safari + Android Chrome both. `lighthouse --emulated-form-factor=mobile` PWA score ≥ 90 before merge.
- Web Push subscriptions are stored per device, not per user — one user with iPad + iPhone + laptop = three subscriptions.
- Per-PR DoD in §6.

## 8.1 — PWA foundation

Branch: `feat/pwa-foundation`

### Goal

`https://app.rapidly.tech` becomes installable as a PWA on iPad
Safari and Android Chrome. Offline shell loads (cached HTML +
CSS + critical JS) even with no connection; data fetches show an
offline state.

### Surfaces

- **Deps:** `cd clients/apps/web && pnpm add @serwist/next serwist`.
- **`next.config.mjs`** — wrap with `withSerwist({...})`. Reuse the existing collab service-worker registration pattern (`clients/apps/web/src/utils/collab/service-worker-registration.ts`).
- **Manifest:** `clients/apps/web/src/app/manifest.ts` (Next.js route-based manifest). Includes:
  - `name`, `short_name` = "Rapidly"
  - `display`: `standalone`
  - `start_url`: `/dashboard`
  - `theme_color` + `background_color` matching emerald palette
  - `icons`: 192/512/1024 + maskable variants
- **Icons:** generate from the existing favicon source. Use `@vite-pwa/assets-generator` or equivalent; commit the generated PNGs.
- **Viewport meta:** in `app/layout.tsx`, set `viewport-fit=cover` + `user-scalable=no` (markup canvas handles its own pinch zoom). `apple-mobile-web-app-capable=yes`.
- **Service worker:** generated by serwist; precaches the app shell and runtime-caches API GETs with stale-while-revalidate. Skip API mutations.
- **Install prompt:** show a "Add to home screen" hint after 3rd visit on a non-installed mobile session. Dismissable.

### Offline behavior

- Static shell + previously-visited routes load from cache.
- API GETs: stale-while-revalidate; show cached data immediately with a "last synced X min ago" badge.
- API mutations: queue locally in IndexedDB; replay on reconnect. UI shows a sync-pending badge.
- Markup boards: already offline-tolerant (Yjs local persistence). Confirm the Yjs `IndexeddbPersistence` adapter is wired — if not, this PR adds it.

### Tests

- Lighthouse PWA audit ≥ 90 in CI (use `lhci` GitHub action).
- Service-worker registration test (extends the existing `service-worker-registration.test.ts`).
- Offline test: load app online, go offline, reload — shell renders, "offline" indicator appears.
- iOS Safari install test (manual): add to home screen, open from icon, navigate.

### Verify

```bash
cd clients/apps/web
pnpm typecheck && pnpm lint && pnpm test
pnpm build && pnpm start
# Browser: Lighthouse PWA score ≥ 90
# iPad Safari: Share → Add to Home Screen → open → confirm standalone
# Android Chrome: install prompt → install → open → confirm standalone
```

## 8.2 — Mobile Markup touch interactions

Branch: `feat/mobile-markup-touch`

### Goal

A field engineer on an iPad can mark up a PDF underlay (from M2.1)
with the same fluency they'd have on desktop with a mouse. Apple
Pencil pressure → stroke width (already wired in the existing
`pressure.ts`). Two-finger pan + pinch-zoom. Long-press to open the
properties sheet (already exists; just needs the touch trigger).

### Surfaces

- `clients/apps/web/src/utils/markup/touch-input.ts` (new) — translates Touch / Pointer events into the canvas's existing input model. Distinguishes:
  - Single finger: draw / drag.
  - Two fingers: pan + pinch-zoom.
  - Long-press (≥ 500 ms, near-zero movement): contextual properties sheet.
  - Three fingers tap: undo (matches Apple Notes muscle memory).
- `Whiteboard/MobilePropertiesSheet.tsx` already exists. Trigger from long-press in this PR.
- **Apple Pencil:** already integrated via `PointerEvent.pressure`; verify in this PR that no regressions slipped in.
- **Palm rejection:** ignore touches that begin while a pen pointer is active.
- **Toolbar:** on viewports < 1024px, collapse the toolbar to a bottom dock (thumbs reach). Existing toolbar component gets a responsive variant.

### CSS

- `touch-action: none` on the canvas (we own all gestures).
- `overscroll-behavior: contain` on the chamber wrapper to stop iOS rubber-band.
- Bottom dock uses `env(safe-area-inset-bottom)` so it doesn't sit on the home indicator.

### Tests

- Unit tests for touch-input.ts gesture recognition (Vitest with synthetic Touch events).
- Playwright with `@playwright/test` mobile device emulator (iPad Pro 11 + Pixel 7) — assert pinch zoom works, long-press opens properties.
- Manual on real iPad + Android tablet (note device in DoD).

### Verify

Manual end-to-end: import a PDF underlay, pinch to zoom into a wall detail, sketch an annotation, long-press for properties, change color, send to a peer — confirm Yjs sync still works.

## 8.3 — Mobile viewer touch nav

Branch: `feat/mobile-viewer-touch`

### Goal

3D model navigation on a tablet using xeokit's
`TouchPickFlyCameraControl` (already shipped in xeokit-sdk; we just
need to wire it). Tap-to-pin places a clash/RFI pin at the touched
world coord.

### Surfaces

- Update `Viewer/ViewerCanvas.tsx` from M3.2: when running on a touch device, swap the default `CameraControl` for `TouchPickFlyCameraControl`. Detect touch via `('ontouchstart' in window)`.
- Two-finger pinch: zoom. Single-finger drag: orbit. Two-finger drag: pan. Three-finger tap: reset view.
- Tap-to-pin (with Clash or RFI pin tool active): xeokit `pick` on the touched canvas position → place the pin.
- Section + measurement tools (M3.4): touch-friendly handles. Section plane drag handles enlarged from 8px to 24px on touch devices.

### Tests

- Touch device detection.
- Tap-to-pin world-coordinate accuracy (within ±10 mm of the desktop equivalent for the same fixture).
- Pinch-zoom doesn't fight the markup chamber's pinch-zoom when a 3D viewport is embedded in a markup board (event ownership conflict — `stopPropagation` inside the viewport's bounding rect).

### Verify

Manual on iPad: open Schependomlaan in viewer, orbit/pan/zoom by touch, tap a beam to place a clash pin, switch to a markup board with that viewport embedded, confirm pin appears at the same beam.

## 8.4 — Mobile site-walk capture

Branch: `feat/mobile-site-walk-capture`

### Goal

The flow that M6.5's "Site walk capture" node exists to consume.
Field engineer opens the app on phone or tablet, taps "Site walk,"
picks a project, captures photos + voice notes, hits Submit, the
backend kicks off the workflow.

### Surfaces

- New route: `clients/apps/web/src/app/(main)/dashboard/[workspace]/site-walk/page.tsx`.
- Components:
  - `SiteWalkCapture/CameraView.tsx` — `MediaDevices.getUserMedia({ video: { facingMode: 'environment' } })` + photo capture (canvas snapshot, JPEG encode).
  - `SiteWalkCapture/VoiceRecorder.tsx` — MediaRecorder API for audio; visual waveform via `Web Audio API`.
  - `SiteWalkCapture/ProjectPicker.tsx` — workspace's projects with offline-cached list.
  - `SiteWalkCapture/UploadQueue.tsx` — IndexedDB queue; uploads with `navigator.connection` aware concurrency.
- Hook: `useSiteWalkSession` orchestrates state across captures.
- Geolocation: `navigator.geolocation.getCurrentPosition()` snapshots lat/lng at each photo. Tagged onto the upload metadata.
- Transcription: client-side via Web Speech API (free, no server cost) — produces a draft transcript the engineer can edit before submit. Fallback to server-side via the M4.4 LLM (Whisper-equivalent) if Web Speech is unavailable.

### Backend

`POST /api/site-walks` — creates a site-walk record, returns presigned upload URLs for photos + audio, queues the M6.5 workflow on completion.

```python
# server/rapidly/projects/site_walk/
class SiteWalk(BaseEntity):
    __tablename__ = "site_walks"
    project_id, captured_by_id, captured_at
    photo_file_ids: Mapped[list[UUID]] = mapped_column(JSONB)
    audio_file_id (nullable)
    transcript: Mapped[str | None] = mapped_column(Text)
    geo_lat (nullable), geo_lng (nullable)
    workflow_run_id (nullable, set after the workflow is kicked off)
```

### Tests

- Photo capture happy path (mocked MediaDevices).
- Voice recorder + Web Speech transcription (mocked SpeechRecognition).
- Offline submit: capture → go offline → submit → reconnect → upload completes.
- Geolocation permission denied → site walk still proceeds without coords.

### Verify

Manual on iPhone + iPad: walk into a room, capture 3 photos, talk for 30 s, edit transcript, submit. Verify the M6.5 workflow runs and creates work items + pins.

## 8.5 — Web Push + phone read-only mode + polish pass

Branch: `feat/web-push-and-phone-readonly`

### Web Push

VAPID push subscriptions per device:

```
POST   /api/push/subscribe        body: { endpoint, p256dh, auth, user_agent }
DELETE /api/push/subscribe/{id}   on logout / unsubscribe
```

```python
# models/push_subscription.py
class PushSubscription(BaseEntity, SoftDeleteMixin):
    __tablename__ = "push_subscriptions"
    user_id
    endpoint (Text), p256dh, auth          # the standard web-push payload triplet
    user_agent                              # for human-readable list "iPad Safari, Pixel 7 Chrome, ..."
    last_used_at
```

Dispatch path: `messaging/notification/workers.py` (existing) adds a
new fan-out target. When a notification is dispatched for a user, we
fan out to (a) in-app, (b) email if configured, (c) **all the user's
push subscriptions**. Payload includes a deep-link URL so tapping
the notification lands in the relevant chamber.

VAPID key generation: one server-side keypair stored in the existing
Fernet-protected secret store. Public key shipped to clients on
boot.

### Phone read-only drawer

A drawer-style nav when viewport < 768px. Surfaces:

- Notifications (the user's unread notifications + push-notification deep links)
- RFIs I'm responsible for (read-only)
- Boards I'm subscribed to (read-only — opens a tap-to-zoom viewer)
- Projects (read-only list)

Markup editing, agent editing, viewer navigation, settings — all
hidden / show a "Open on tablet or larger" message.

Implementation: gate the existing dashboard layout on viewport;
swap to `<PhoneReadOnlyLayout />` below 768px.

### Polish pass

A grab-bag of M1–M7 cleanup items. Each is a sub-commit in this PR.
Track in a checklist in the PR description; non-negotiable for the
PR to merge:

- [ ] Loading skeletons added wherever a network call lacks one.
- [ ] Error boundaries added at each chamber's top-level layout.
- [ ] 404 page styled to match the suite.
- [ ] All Playwright specs marked `test.describe.parallel` where safe.
- [ ] OpenAPI client snapshot test in CI to catch silent drift.
- [ ] Empty-state illustrations for: empty board, empty viewer (no models), empty workflows list, empty RFIs.
- [ ] Dark-mode pass on every chamber (the design system already supports it; M5 + M6 surfaces probably regressed).
- [ ] Performance budget: TTI ≤ 3 s on a mid-tier Android (Pixel 6a baseline).

### Tests

- Web Push subscription round-trip; dispatch fan-out hits the registered endpoint (mocked).
- Phone-mode rendering: at 360 × 800 viewport, only the read-only drawer is mounted.
- Markup chamber rendering at < 768px: shows the "open on a larger screen" message.

### Verify

Manual:
- iPhone Safari: subscribe to push, trigger a notification, confirm it arrives on the lock screen.
- iPhone: open `/dashboard`, see the drawer + read-only surfaces, attempt to open a markup board, see the "tablet or larger" message.
- iPad: regular full UX still works.
